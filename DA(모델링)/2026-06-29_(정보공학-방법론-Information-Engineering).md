## 1. 핵심 요약
James Martin이 체계화한 **데이터 중심의 기업 전체 정보시스템 개발 방법론**. ISP → BAA → BSD → SC(구축)의 4단계 폭포수 방식으로 진행하며, ERD·DFD 등 구조적 산출물을 단계별로 도출한다. 대규모 기업 시스템에 적합하나 웹·분산 환경에 적용이 어렵고 산출물이 복잡하다는 한계가 있다.

## 2. 정보공학 방법론 개요

### 가) 정의
- **기업 전체(또는 주요부분)의 계획·분석·설계·구축에 정형화된 기법들을 통합 적용하는 데이터 중심 방법론**
- James Martin이 비즈니스 시스템 개발을 공학적으로 접근하기 위해 체계화

### 나) 특징

| 특징 | 내용 |
|------|------|
| **기업 중심** | 적용 대상이 기업의 비즈니스 시스템. 전략경영 지원을 위한 SIS(Strategic Information System) 초점 |
| **ISP 중심** | 경영층의 요구와 견해를 시스템에 반영하는 정보전략계획이 핵심 |
| **데이터 중심** | 수시로 변하는 업무 프로세스 대신, 잘 변하지 않는 데이터 기반 설계 → 유지보수 절감 |
| **분할과 정복** | Top-Down 방식. 문제영역 → 업무영역(BA) → 업무시스템(BS) → 구현시스템(IS) |
| **공학적 접근** | Case Tool, 소스코드 자동 생성, 모듈화, 다이어그램 사용 |
| **사용자 참여** | 초기 단계부터 사용자의 적극적 참여와 피드백 반영 |

### 다) 장/단점

| 장점 | 단점 |
|------|------|
| 경쟁우위 확보의 전략적 기회 식별 및 방안 제공 | 효과를 위해 장기간 소요 |
| 일관성 있고 통일된 정보시스템 구축 가능 | 소규모 자동화 사업영역에는 부적합 |
| 시스템의 장기적인 진화·발전 허용 | 특정 사업영역 독립 시스템 개발에 부적합 |
| 데이터 중심으로 업무절차·환경변화에 유연 | |

## 3. 구성 및 단계별 명세

### 가) 4단계 구성

![[Pasted image 20260626000105.png]]

| 절차 | 설명 |
|------|------|
| **정보전략계획 (ISP)** | 경영전략·관련조직·업무자료 거시적 분석, 현행시스템 평가 |
| **업무영역 분석 (BAA)** | 데이터 모델링(ERD), 프로세스 모델링(DFD) |
| **업무시스템 설계 (BSD)** | 업무절차 정의, Presentation 설계, 분산설계 |
| **시스템 구축 (SC)** | 응용 프로그램 작성 |

### 나) ISP 단계 상세

전사적인 기업모형(청사진)을 설계하는 단계.

| 단계 | 특징 |
|------|------|
| 경영전략 분석 | 기업 내·외부 환경분석 및 기업비전·전략 도출 |
| 현행 업무 프로세스 분석 | 문제점 및 개선방안 도출 |
| 현 시스템 분석·평가 | 현 시스템 문제점 도출 평가 후 개선방안 |
| 아키텍처 개발 | 후속작업 지원을 위한 프로세스·데이터 모델 개발 |
| 전략 계획 | 프로젝트 정의 및 우선순위 부여 |

### 다) BAA 단계 산출물

ISP(정보전략계획) 과정에서 수집된 정보를 세부적으로 확장.

| 산출물 | 내용 |
|--------|------|
| 데이터 모델 다이어그램 | ISP에서 만들어진 ERD를 상세하게 확장 |
| 프로세스 분할 다이어그램 | 업무영역 내 기능들을 프로세스로 분할한 트리구조 분할도 |
| 프로세스 의존 다이어그램 | 서로 다른 프로세스간 의존관계 표현 |
| 프로세스/데이터 매트릭스 | 프로세스와 데이터간 행위를 매트릭스로 표현 |

### 라) BSD 단계 도구

| 구분 | 내용 |
|------|------|
| 엔티티-관계 다이어그램 (ERD) | 전 과정에 걸쳐 추상화된 데이터간 함수적 관계를 그림으로 표현 |
| 분할 다이어그램 | 최상위 전략수립 단계에서 하위단계로 진행하며 기능·프로세스·프로시저 분할도 표현 |
| 액션 다이어그램 | Case Tool로 분할 다이어그램을 변환, 향후 실행 가능한 코드로 전환 |
| 의존 다이어그램 | 프로세스 상호간 연관(분할 다이어그램의 우선순위 약점 보완) |
| 데이터 흐름도 (DFD) | 각 프로시저의 입출력 데이터 흐름을 그림으로 표현 |
| 대화구조 표현 | 계층적 메뉴(액션다이어그램), 수평 대화(Dialog Flow Diagram) |
| 자료구조 다이어그램 | BAA의 데이터 모델을 해당 DBMS에 맞게 다이어그램 생성 |

## 4. 방법론 비교 (IE vs OO vs CBD)

| 항목 | 정보공학 (IE) | 객체지향 (OO) | CBD |
|------|-------------|-------------|-----|
| 주요관점 | 데이터 중심 | 객체 중심 | 컴포넌트 중심 |
| SDLC 모델 | 폭포수 모델 | 반복적 개발 모델 | 반복적 개발 모델 |
| 주요기술 | PC 보급 확산 | 인터넷 이용 확산 | 비즈니스 기능 단위 컴포넌트 |
| 주요기법 | 데이터모형(ERD), ISP, BRP, 분산설계 | 객체모형, 위험관리, 반복·점진적 개발, 리팩토링 | 컴포넌트 명세·추출·연동·설계 |
| 장점 | 데이터 관점 정규화, 단계별 산출물 명확, 대규모 프로젝트 적합 | 경험 없는 프로젝트에 유용, 재사용성 증대, 의사소통 유리 | 컴포넌트 조립 개발, 생산성 향상, Time To Market 실현 |
| 단점 | 웹·분산·컴포넌트 환경 적용 어려움 | 대규모 프로젝트 적용 어려움, 성숙한 기술자 필요 | 조직별 컴포넌트 기술 수준 차이, 업종별 컴포넌트 부족 |

## 5. ISP vs EAP 비교

| 구분 | ISP | EAP |
|------|-----|-----|
| 수행 목적 | IT전략 수립, 과제 도출 | 전사 IT 가시화, 상호운영성 확보 |
| 상세 수준 | 정보공학 기반 개괄적 시스템 계획 | 아키텍처 기반 상세 수준 계획 |
| 도출 과정 | 컨설턴트 직관적 판단, Business 전략 분석 의존 | 모델링 기법, 다양하고 객관적인 준거 자료 |
| 활용 | 기술·환경 변화 수정 어려움, 일회성 활용 | 기업 통합(EA), EAMS를 통한 지속적 관리 |
| 산출물 | 전략과제 (아키텍처 구체 산출물 없음) | EA비전·프레임웍, EA정보(BA/AA/DA/TA), 이행계획 |
| 수행 횟수 | N번 (반복 수행 안 함) | 1번 |
| 변경관리 | N | Y |
| 관리 시스템 | 없음 | EAMS |

## 6. 정보공학 방법론의 한계점

- **경직된 구조**: 생명주기에 따라 이전 단계의 완성을 토대로 다음 단계 진행 → 폭포수 모형처럼 잘못된 작업을 거스르기 어려움
- **복잡한 산출물**: 복잡하고 종류가 많은 산출물·절차를 따르기 어려움
- **고비용 도구**: 값비싼 CASE Tool 도입과 까다로운 사용법 숙지 필요
