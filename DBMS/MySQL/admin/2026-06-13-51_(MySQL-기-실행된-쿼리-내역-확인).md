## 1. 핵심 요약

Performance Schema의 `events_statements_*` 컨슈머를 켜면 특정 세션(`processlist_id`)이 **이미 실행한 쿼리 이력**과 현재 수행문을 추적할 수 있습니다.
`SHOW ENGINE INNODB STATUS` / `SHOW PROCESSLIST` / `INNODB_TRX`로 대상 스레드를 특정한 뒤 `events_statements_history_long`에서 SQL_TEXT를 조회합니다.

---

## 2. 사전 설정 — history_long 컨슈머 활성화

```sql
-- 현재 상태 확인
SELECT * FROM performance_schema.setup_consumers
 WHERE name LIKE 'events_statements_%';
-- events_statements_history_long 이 NO면 OFF → ON 필요

SHOW VARIABLES LIKE 'performance_schema_events_statements_history%';
-- history_long_size=10000, history_size=10
```

> `events_statements_history_long`이 켜져야 장기 이력 추적 가능.

---

## 3. 대상 세션(thread/processlist) 특정

```sql
SHOW ENGINE INNODB STATUS\G   -- TRANSACTIONS 섹션의 MySQL thread id 확인
SHOW PROCESSLIST;             -- Sleep 상태로 오래 대기하는 세션 Id 확인
SELECT * FROM information_schema.INNODB_TRX\G  -- trx_mysql_thread_id, trx_rows_modified
```

> 예: ACTIVE 32초인 트랜잭션의 `trx_mysql_thread_id=56604` 특정. `trx_query=NULL`이라도 undo log entries로 DML 실행을 추정.

---

## 4. 쿼리 이력 조회

```sql
-- 마지막 수행 쿼리 (현재)
SELECT esc.SQL_TEXT
FROM performance_schema.events_statements_current esc
  JOIN performance_schema.threads t ON t.thread_id = esc.thread_id
WHERE t.processlist_id = 56604;

-- 수행 이력 (history_long, 실행시간 포함)
SELECT esh.THREAD_ID, t.processlist_id,
       sys.format_time(esh.timer_wait) AS 'wait time',
       esh.SQL_TEXT
FROM performance_schema.events_statements_history_long esh
  JOIN performance_schema.threads t ON t.thread_id = esh.thread_id
WHERE t.processlist_id = 56604
ORDER BY TIMER_END ASC LIMIT 10;
```

> `events_statements_current`는 현재 수행문, `events_statements_history_long`은 과거 이력.
