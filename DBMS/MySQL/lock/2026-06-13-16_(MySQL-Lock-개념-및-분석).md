## 1. 핵심 요약

MySQL의 락은 **행 단위 락(Record/Gap/Next-Key)**과 **테이블 단위 락(MDL, Intent)**으로 구분됩니다.
DDL 실패의 대부분은 **MDL_EXCLUSIVE 또는 IX 락**이 미해제된 세션 때문입니다.
Performance Schema의 data_locks 테이블로 락 홀더 세션을 식별하고, SQL_TEXT로 원인 쿼리를 찾습니다.

---

## 2. 락 타입 및 모드

| LOCK_TYPE | LOCK_MODE | 용도 | 예시 |
|-----------|-----------|------|------|
| RECORD | X | 레코드 쓰기 | UPDATE, DELETE |
| RECORD | S | 레코드 읽기 | SELECT...LOCK IN SHARE MODE |
| TABLE | IX | Intent Exclusive (DML 진행중) | INSERT, UPDATE 열려있음 |
| TABLE | IS | Intent Shared | SELECT 진행중 |
| TABLE | X | 테이블 쓰기 | LOCK TABLES... WRITE |
| TABLE | S | 테이블 읽기 | LOCK TABLES... READ |
| TABLE | MDL_EXCLUSIVE | DDL 전용 | ALTER TABLE, DROP, TRUNCATE |
| TABLE | MDL_SHARED | 읽기 공유 | SELECT (기본) |

**핵심**: DDL 실패 → **MDL_EXCLUSIVE** 또는 **IX 락 미해제** 확인

---

## 3. 락 홀더 조회 (Performance Schema)

```sql
SELECT
  dl.OBJECT_SCHEMA,
  dl.OBJECT_NAME,
  dl.LOCK_TYPE,
  dl.LOCK_MODE,
  t.PROCESSLIST_ID AS pid,
  t.PROCESSLIST_USER AS user,
  t.PROCESSLIST_HOST AS host,
  es.SQL_TEXT AS current_sql
FROM performance_schema.data_locks dl
JOIN performance_schema.threads t ON t.thread_id = dl.thread_id
LEFT JOIN performance_schema.events_statements_current es ON es.thread_id = dl.thread_id
WHERE dl.OBJECT_SCHEMA = 'target_db'
  AND dl.OBJECT_NAME = 'target_table'
ORDER BY dl.LOCK_MODE DESC;
```

---

## 4. 원인 쿼리 판단

**Step 1**: LOCK_MODE가 IX 또는 MDL_EXCLUSIVE인 세션 식별
```sql
-- IX 락 홀더 찾기
SELECT * FROM performance_schema.data_locks
WHERE LOCK_MODE IN ('IX', 'MDL_EXCLUSIVE');
```

**Step 2**: 해당 세션의 현재 SQL 확인
```sql
SELECT SQL_TEXT FROM performance_schema.events_statements_current
WHERE THREAD_ID = (위에서 찾은 thread_id);
```

**Step 3**: 트랜잭션 상태 확인
```sql
SELECT * FROM information_schema.innodb_trx
WHERE THREAD_ID = (위의 thread_id);
-- trx_state = RUNNING → 트랜잭션 열려있음 (COMMIT 미실행)
```

## 5. 락 해제 방법

### 5.1 정상 해제
```sql
-- 해당 세션에서
COMMIT;
```
