## 1. 핵심 요약

300GB 이상 테이블 DDL은 가급적 회피(child 테이블+join 모델링). 작업 전 **lock_wait_timeout·innodb_online_alter_log_max_size(1G→10G)·innodb_adaptive_hash_index=OFF** 를 설정하고, **Free Local Storage(tmpdir)·Freeable Memory** 를 모니터링한다. 대표 실패 케이스 5종(로컬스토리지 부족, online dml buffer 부족, MDL timeout, 칼럼+즉시인덱스 카디널리티1, duplicate apply)을 숙지한다.

---

## 2. 작업 전 필수 설정

```sql
set session lock_wait_timeout = 120;   -- 단, 서비스 쿼리 메타락 대기 밀림 주의(롱쿼리 없으면 5초로 충분)
innodb_online_alter_log_max_size = 10737418240;  -- 1G→10G
innodb_adaptive_hash_index = OFF;      -- DDL 시 AHI 비우기 부하로 DB 행 걸림 방지
```

## 3. 작업시간·자원 산정

| 작업 | 산정 | 비고 |
|------|------|------|
| 인덱스 추가 | data 1GB당 30초 | tmpdir에 테이블 크기만큼 Free Local Storage 필요 |
| 컬럼 추가 등 inplace | (data+index) 1GB당 1분 | datadir(클러스터볼륨) 사용 |
| 파티션 테이블 | 위 ×1.5 | 파티션 수 영향 |

> Freeable Memory는 작업 종료 후에도 반환 안 됨 → 부족하면 DB 재기동.

## 4. 실패 케이스 5종

| # | 케이스 | 원인·증상 |
|---|--------|-----------|
| 1 | Free Local Storage 부족 | 인덱스 추가 시 tmpdir 사용. `ERROR 1878 Temporary file write failure` (add column rebuild은 datadir라 무관) |
| 2 | online dml buffer 부족 | `innodb_online_alter_log_max_size` 초과. **막바지 apply 시점에 실패**(가장 아픔) |
| 3 | metadata lock timeout | 롱트랜잭션이 MDL 보유 → DDL 대기 + 후속 쿼리 모두 대기 → 서비스 장애. `ERROR 1205` |
| 4 | 칼럼+즉시 인덱스 | 카디널리티 1(통계 미갱신) → 풀스캔. DDL과 쿼리배포 텀 길게 or 복합인덱스 |
| 5 | Duplicate apply | DDL 중 DML 로그버퍼 apply에서 duplicate error → LOCK TABLES 또는 pt-osc 사용 |

## 5. 진행 모니터링

```sql
UPDATE performance_schema.setup_instruments SET ENABLED='YES' WHERE NAME LIKE 'stage/innodb/alter%';
UPDATE performance_schema.setup_consumers SET ENABLED='YES' WHERE NAME LIKE 'events_stages%';
-- WORK_COMPLETED/WORK_ESTIMATED (page 단위 작업량)
```
