# MySQL GTID 기반 복제

- **카테고리**: #DBMS #MySQL #Replication #GTID
- **작성일**: 2026-06-13
- **참조 원본**: [[2026-06-12-02. replication with global transaction identifiers - BigDataTeam.md]]

## 1. 핵심 요약

GTID (Global Transaction ID)는 **Master-Slave 복제 토폴로지 전체에서 유일한 트랜잭션 식별자**입니다.
Binary Log Position 방식의 수동 위치 관리를 제거하고, 자동으로 위치를 결정하는 **MASTER_AUTO_POSITION** 방식을 제공합니다.
각 트랜잭션이 커밋되면 MySQL이 자동으로 GTID를 할당하고, Slave는 이미 적용한 GTID를 Skip하여 **중복 실행 방지**합니다.

---

## 2. GTID 기본 개념

### 2.1 GTID 형식

```
GTID = source_uuid:transaction_id
```

**예시**:
- `3E11FA47-71CA-11E1-9E33-C80AA9429562:1-5` → 1번부터 5번까지의 트랜잭션
- `3E11FA47-71CA-11E1-9E33-C80AA9429562:1-3:11:47-49` → 1-3번, 11번, 47-49번 트랜잭션

### 2.2 GTID 특징

- **Universally Unique**: 복제 토폴로지 전체에서 중복 없음
- **자동 할당**: 각 트랜잭션 커밋 시 자동 생성
- **추적 용이**: 로그 파일/위치 없이 트랜잭션 추적 가능
- **Slave 간 공유**: 어느 Slave에서든 적용된 트랜잭션 GTID 확인 가능

## 3. GTID 저장 메커니즘

### 3.1 mysql.gtid_executed 테이블

```sql
CREATE TABLE gtid_executed (
  source_uuid CHAR(36) NOT NULL,
  interval_start BIGINT(20) NOT NULL,
  interval_end BIGINT(20) NOT NULL,
  PRIMARY KEY(source_uuid, interval_start)
);
```

**역할**:
- Binary Log에 저장된 것을 제외한 모든 트랜잭션 GTID 보관
- Binary Log가 손실되어도 GTID 정보 유지
- 자동 압축 (기본 1000개 단위)

### 3.2 gtid_executed 시스템 변수

```sql
SHOW VARIABLES LIKE 'gtid_executed';
-- 출력: 3E11FA47-71CA-11E1-9E33-C80AA9429562:1-100
```

- Master & Slave 모두 **커밋된 모든 트랜잭션 GTID** 포함
- Binary Log에 저장된 최신 정보도 포함

## 4. GTID 생명주기

### 4.1 Step 1: Master에서 GTID 할당

```
트랜잭션 실행 → COMMIT → GTID 할당 (uuid:seq) 
→ Binary Log에 기록 → gtid_executed에 추가
```

### 4.2 Step 2: Slave로 전송

```
Master Binary Log → Slave Relay Log → GTID 읽음 → gtid_next 설정
```

### 4.3 Step 3: Slave에서 처리

```
1. gtid_next의 GTID가 gtid_executed에 있는지 확인
2. 없으면 → 트랜잭션 실행 (새 GTID 생성 X, gtid_next 값 사용)
3. 있으면 → 자동 SKIP (중복 실행 방지)
```

### 4.4 Step 4: Slave에 기록

```
Binary Logging ON:  binary log에 기록
Binary Logging OFF: mysql.gtid_executed 테이블에 직접 기록
```

## 5. GTID 설정 (cold start)

### 5.1 Master & Slave my.cnf 설정

```ini
[mysqld]
# GTID 활성화
gtid_mode=ON
enforce_gtid_consistency=ON

# Master 필수
log-bin=mysql-bin
server-id=1

# Slave (선택, chain replication 시 필수)
log-bin=mysql-bin
log_slave_updates=ON
server-id=2
```

### 5.2 서버 시작

```bash
# 기존 복제가 실행 중이면 먼저 동기화
SET GLOBAL READ_ONLY = ON;  # Master와 Slave 모두 실행

# 서버 종료 후 위 설정으로 재시작
mysqld_safe --defaults-file=/path/to/my.cnf
```

## 6. Slave 자동 위치 설정 (MASTER_AUTO_POSITION)

### 6.1 Slave 설정

```sql
STOP SLAVE;

CHANGE MASTER TO
  MASTER_HOST='master_ip',
  MASTER_USER='repl_user',
  MASTER_PASSWORD='password',
  MASTER_AUTO_POSITION=1;    -- Binary Log 파일/위치 불필요

START SLAVE;

SHOW SLAVE STATUS\G
```

### 6.2 자동 위치 결정 원리

```
초기 Handshake:
  ↓
Slave가 Master에 자신의 gtid_executed 전송
  ↓
Master가 자신의 gtid_executed와 비교
  ↓
차이나는 부분만 전송 시작
  ↓
Slave가 자동으로 Position 결정
```

## 7. GTID 시스템 변수

| 변수 | 설명 | 기본값 |
|------|------|--------|
| `gtid_mode` | GTID 활성화 (ON/OFF/OFF_PERMISSIVE/ON_PERMISSIVE) | OFF |
| `enforce_gtid_consistency` | GTID 일관성 강제 | OFF |
| `gtid_executed` | 실행된 GTID Set (조회용) | 읽기 전용 |
| `gtid_purged` | Binary Log에 없지만 실행된 GTID | 설정 가능 |
| `gtid_next` | 다음 트랜잭션에 사용할 GTID | AUTOMATIC |

### 7.1 gtid_next 활용

```sql
-- 특정 GTID 건너뛰기
SET SESSION gtid_next='3E11FA47-71CA-11E1-9E33-C80AA9429562:100';
BEGIN; COMMIT;
SET SESSION gtid_next='AUTOMATIC';
```

## 11. GTID 초기화

### 11.1 RESET MASTER

```sql
-- Master에서만 실행
-- ⚠️ 모든 Binary Log + GTID 정보 초기화
RESET MASTER;

-- 초기화되는 항목:
-- - gtid_purged → 빈 문자열
-- - gtid_executed → 빈 문자열
-- - mysql.gtid_executed 테이블 → 레코드 삭제
-- - Binary Log 파일 삭제
```

### 11.2 RESET SLAVE ALL

```sql
-- Slave에서만 실행
-- Master 채널 정보 + GTID 정보 초기화
RESET SLAVE ALL;
```

## 12. Binary Log Position vs GTID 비교

| 항목 | Position 기반 | GTID 기반 |
|------|--------------|----------|
| **위치 관리** | 수동 (파일/position) | 자동 |
| **Slave 추가** | 복잡 (정확한 position 필요) | 간단 (MASTER_AUTO_POSITION=1) |
| **장애 복구** | 수동 위치 재설정 필요 | 자동 위치 결정 |
| **중복 실행 방지** | 없음 | GTID Skip으로 자동 방지 |
| **Multi-Master** | 지원 제한 | 우수한 지원 |

## 13. 권장 사항

✅ **GTID 사용 권장**:
- 새로운 복제 구성
- Slave 추가/변경 빈번
- HA/Failover 운영
- MySQL 5.7+

⚠️ **Position 기반 고려**:
- 레거시 시스템 호환성
- 특별한 이유로 GTID 비활성화 필요
