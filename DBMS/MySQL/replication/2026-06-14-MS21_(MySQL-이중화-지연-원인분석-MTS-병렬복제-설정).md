## 1. 핵심 요약

`Seconds_Behind_Master`가 증가할 때 **IO Thread 지연**인지 **SQL Thread 지연**인지 먼저 구분한다. SQL Thread 지연이면 `slave_parallel_workers`와 `slave_parallel_type`을 늘려 MTS 병렬 처리로 해결한다. DB가 여러 개면 `DATABASE` 타입, 단일 DB면 `LOGICAL_CLOCK` 타입이 효과적이다.

---

## 2. 지연 유형 판별

```sql
SHOW MASTER STATUS;             -- master 현재 binlog 위치
SHOW SLAVE STATUS\G             -- IO Thread / SQL Thread 위치 비교
```

| 필드 | 의미 |
|------|------|
| `Read_Master_Log_Pos` | IO Thread가 master에서 읽은 위치 |
| `Exec_Master_Log_Pos` | SQL Thread가 실행한 위치 |
| `Slave_SQL_Running_State` | 현재 실행 중인 DML 종류 |

→ `Read_Master_Log_Pos ≈ master pos` + `Exec < Read` → **SQL Thread 지연** (대량 DELETE 등).

## 3. 해결: MTS 병렬 복제 설정

### Case 1 — DB 여러 개 (DATABASE 타입)
```sql
SET GLOBAL slave_parallel_workers = 2;  -- DB 수와 동일하게
STOP SLAVE; START SLAVE;
-- slave_parallel_type: DATABASE (기본)
```

### Case 2 — DB 단일 (LOGICAL_CLOCK 타입 권장)
```sql
STOP SLAVE SQL_THREAD;
SET GLOBAL slave_parallel_type = LOGICAL_CLOCK;
SET GLOBAL slave_parallel_workers = 4;
START SLAVE SQL_THREAD;
```

> `LOGICAL_CLOCK`: 동일 master commit 그룹을 병렬로 적용 → 단일 DB에서도 효과.

## 4. 모니터링 스크립트

```bash
while :; do
  LAG=$(mysql -uroot -p*** -e "show slave status \G;" | grep -i behind | cut -d':' -f2 | tr -d ' ')
  echo $(date "+%Y%m%d %H:%M:%S") "---$LAG"
  if [ $LAG -gt 10 ]; then
    mysql -e "show slave status \G;" >> repevent.log
  fi
  sleep 3
done
```
