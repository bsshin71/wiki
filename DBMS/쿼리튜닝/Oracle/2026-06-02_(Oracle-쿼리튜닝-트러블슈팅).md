## 1. 핵심 요약
- **Snapshot too old**: 불필요한 잦은 커밋, 오래 걸리는 쿼리와 대량 트랜잭션 혼재, Nested Loop 반복 블록 방문이 주요 원인.
- **함수 병목**: SELECT 절 사용자 함수는 결과 건수에 상관없이 조건 만족 전체 레코드만큼 recursive call 발생 → Inline View, JOIN, 스칼라 서브쿼리 캐싱으로 대체.

## 2. 상세 설명

### Snapshot Too Old 발생 가능성 줄이기

ORA-01555 에러. Undo 세그먼트가 오래된 이미지를 덮어써서 Read Consistency 실패.

| 원인 | 해결 방법 |
|------|-----------|
| 불필요하게 잦은 COMMIT | COMMIT 횟수 줄이기 |
| Fetch Across Commit | ANSI 표준 준수: COMMIT 전에 열린 커서 Fetch 금지. 불가피하면 COMMIT 횟수 최소화 |
| 트랜잭션 폭주 시간대에 장시간 쿼리 혼재 | 실행 시간대 분리 |
| 대용량 테이블 전체 처리 | 범위 분할(페이지 처리)로 재설계 → Snapshot too old + 재시작 포인트 확보 |
| Nested Loop에서 같은 블록 반복 방문 | 조인 메소드 변경 또는 Full Table Scan 유도 |
| ORDER BY 부재로 Fetch 장시간 | ORDER BY 강제 삽입 → Temp 세그먼트 일괄 저장 후 반복 블록 방문 제거 |
| Delayed 블록 클린아웃 의심 시 | 대량 UPDATE 후 Full Scan 실행: `SELECT /*+ FULL(t) */ COUNT(*) FROM table_name t` |

### 함수 부하가 병목인 쿼리 튜닝 (Oracle / Tibero)

**문제 패턴**: SELECT 절에 사용자 함수 사용
```sql
-- 문제: 조건 만족 전체 행 수만큼 JISA_NAME() 함수 호출
SELECT ..., JISA_NAME(A.JISA_code) 지사이름
FROM 자금지원이력 A
WHERE A.기간 BETWEEN sysdate-10/24/60 AND sysdate
GROUP BY A.region;
```

**튜닝 방법 4가지:**

**1) Inline View로 최종 결과에서만 함수 호출**
```sql
SELECT ..., JISA_NAME(jisa_code) 지사이름
FROM (
  SELECT /*+ NO_MERGE */ A.region, A.jisa_code, ...
  FROM 자금지원이력 A
  WHERE A.기간 BETWEEN sysdate-10/24/60 AND sysdate
  GROUP BY A.region
);
```

**2) DECODE / CASE 문으로 함수 대체** (함수 로직이 계산식인 경우)

**3) INNER JOIN / OUTER JOIN으로 변환**
```sql
SELECT B.JISA_NAME
FROM 자금지원이력 A, 지사이름 B
WHERE A.jisa_code = B.jisa_code
AND A.기간 BETWEEN sysdate-10/24/60 AND sysdate
GROUP BY A.region;
```

**4) 스칼라 서브쿼리 캐싱 (Oracle)**
```sql
-- Oracle: FROM DUAL로 캐싱 효과
SELECT (SELECT JISA_NAME(A.JISA_code) FROM dual) 지사이름, ...
```

**5) DETERMINISTIC 함수 선언 (Oracle)**
```sql
CREATE OR REPLACE FUNCTION JISA_NAME(p_jisacode varchar)
RETURN varchar DETERMINISTIC  -- 동일 입력 → 동일 출력 보장 → 캐싱
...
```

> ⚠️ Tibero: `FROM dual` 구문 없이도 plan상 sub쿼리 caching 표시되나 실제 캐싱 여부 불확실. 검증 필요.
