## 1. 핵심 요약

오라클 SQL 튜닝의 핵심은 **"원인을 정확히 진단하고, 불필요한 I/O(Block 액세스)를 줄이는 것"** 입니다. 단편적인 팁(힌트 무작정 쓰기)보다 옵티마이저의 관점에서 접근해야 성공 확률이 높습니다. AUTOTRACE나 `DBMS_XPLAN.DISPLAY_CURSOR`로 실제 수행된 Real 실행계획과 I/O 수치를 반드시 확인합니다.

---

## 2. 진단 및 실행계획 분석 (가장 중요)

튜닝의 시작은 추측이 아니라 데이터에 기반한 진단입니다.

**진행 순서:**
1. SQL의 실제 실행 계획(Execution Plan) 확인
2. 각 단계별 Buffers(Logical I/O) 수치 확인
3. 대량의 데이터를 소비하는(병목이 생기는) 지점 식별

> ⭐ **실무 팁**: AUTOTRACE나 `DBMS_XPLAN.DISPLAY_CURSOR`를 활용해 **실제 수행된 Real 실행계획**과 I/O 수치를 보세요. 단순 예상 실행계획은 옵티마이저의 착각일 수 있습니다.

---

## 3. 인덱스(Index) 스캔 최적화

실무 성능 저하의 80%는 잘못된 인덱스 사용이나 인덱스 부재에서 옵니다.

### 좌변(컬럼) 가공 금지

```sql
-- ❌ 인덱스 못 탐
WHERE TO_CHAR(REG_DATE, 'YYYYMMDD') = '20260614'

-- ✅ 인덱스 사용 가능
WHERE REG_DATE >= TO_DATE('20260614', 'YYYYMMDD')
  AND REG_DATE <  TO_DATE('20260615', 'YYYYMMDD')
```

### 묵시적 형변환 주의

컬럼이 VARCHAR2인데 조건절에 숫자(`WHERE EMP_NO = 1234`)를 넣으면 오라클이 내부적으로 `TO_NUMBER(EMP_NO)`로 변환해 인덱스를 타지 못합니다. **타입을 반드시 일치**시키세요.

### 복합 인덱스 구성 전략

조건절에 많이 쓰이는 컬럼 중:
- **`=` (동등) 조건 컬럼** → 인덱스 **앞쪽**에 배치
- **범위 조건(LIKE, BETWEEN, >, <) 컬럼** → 인덱스 **뒤쪽**에 배치

→ 스캔 범위를 최소화할 수 있습니다.

> **실무 팁**: NL Join이 느리다면 드라이빙 테이블의 추출 건수가 너무 많지 않은지, 드리븐 테이블의 조인 컬럼에 인덱스가 빠졌는지 확인하세요.

---

## 4. 조인(Join) 방식의 올바른 선택

테이블 간의 데이터 양과 성격에 따라 조인 방식을 튜닝해야 합니다.

| 조인 방식                      | 특징                  | 실무 적용 기준                                  |
| -------------------------- | ------------------- | ----------------------------------------- |
| **NL Join** (Nested Loops) | 순차적 접근, 인덱스 필수      | OLTP성(실시간 화면), 처리 건수가 적고 인덱스가 잡혀있을 때 유리   |
| **Hash Join**              | 메모리(PGA)에 해시 테이블 생성 | 대량 데이터 조인, 배치(Batch) 업무, 인덱스 없을 때 매우 유리   |
| **Sort Merge Join**        | 양쪽 테이블 정렬 후 조인      | 대량 데이터 비교 시 사용, 최근 인메모리 처리에는 Hash Join 선호 |

---

## 5. 서브쿼리(Subquery) 및 쿼리 재작성

### EXISTS vs IN

최근 오라클(12c, 19c 이상)은 옵티마이저가 쿼리 변환(Query Transformation)을 통해 자동으로 서브쿼리를 Unnesting(조인문으로 변경)합니다. 메인 테이블 조건으로 필터링이 크게 이뤄진 후 소량만 서브쿼리와 비교할 때는 `FILTER` 방식보다 `HASH JOIN SEMI` 등으로 풀리도록 `/*+ UNNEST */` 힌트가 유리할 때가 있습니다.

### OR 조건 지양 → UNION ALL로 분리

```sql
-- ❌ 인덱스 태우기 어려움
WHERE A = 1 OR B = 2

-- ✅ 각각 인덱스를 타게 분리
SELECT ... WHERE A = 1
UNION ALL
SELECT ... WHERE B = 2
```

### 분석 함수(Window Function) 활용

대량 데이터를 Self Join하여 순위나 누적 합을 구하는 구조라면, `ROW_NUMBER()`, `RANK()`, `LEAD()`, `LAG()` 등의 분석 함수를 사용하여 **테이블을 단 한 번만 읽도록** 재작성하세요.

## 6. Full Table Scan vs Index Scan 체크

다음 항목을 실행계획에서 확인합니다:

- **Full Table Scan vs Index Scan**: 인덱스를 타야 할 곳에서 전체 테이블을 읽고 있지 않은가?
- **Join 방식**: 데이터 양에 맞는 적절한 조인(NL·Hash·Merge Join)이 일어나는가?
- **Cost & Cardinality**: 오라클 옵티마이저가 예상한 건수와 실제 건수가 비슷한가?

## 7. 데이터 조회 범위 최소화

### SELECT * 절대 금지

필요한 컬럼만 명시하세요. 불필요한 컬럼은 네트워크 부하와 오라클 메모리(데이터 버퍼 캐시) 낭비를 초래하고, **Index Only Scan**이 가능한 쿼리를 테이블까지 읽게 만듭니다.

### 통계정보(Statistics) 최신화

쿼리나 인덱스에 문제가 없는데도 옵티마이저가 이상한 실행계획을 세운다면, 통계정보가 오래된 것일 수 있습니다.

```sql
EXEC DBMS_STATS.GATHER_TABLE_STATS(ownname=>'SCHEMA', tabname=>'TABLE_NAME');
```

## 8. 힌트(Hint) 사용

옵티마이저가 엉뚱한 경로를 선택할 때 개발자가 직접 경로를 지정합니다.

```sql
/*+ INDEX(A EMP_IX01) */     -- A 테이블에 EMP_IX01 인덱스 강제 적용
/*+ USE_HASH(B) */            -- B 테이블과 해시 조인 유도
/*+ UNNEST */                 -- 서브쿼리를 조인으로 변환 유도
```

### 페이징 처리 (부분 범위 처리)

오라클 12c 이상이라면 `OFFSET N ROWS FETCH NEXT M ROWS ONLY` 구문을 사용하여 필요한 만큼만 끊어서 읽어와야 시스템 부하를 막을 수 있습니다.

## 9. 인덱스를 제대로 타게 만드는 작성법

### 인덱스 컬럼을 변형하지 마세요

```sql
-- ❌ 나쁜 예 (인덱스 사용 불가)
WHERE TO_CHAR(join_date, 'YYYYMMDD') = '20260614'

-- ✅ 좋은 예 (값을 가공)
WHERE join_date = TO_DATE('20260614', 'YYYYMMDD')
```

> 이유: 조건절 좌변(컬럼)을 가공하면 인덱스를 사용할 수 없습니다. **값(우변)을 가공**해야 합니다.

### 묵시적 형변환(Implicit Conversion) 피하기

```sql
-- ❌ 나쁜 예 (emp_no가 VARCHAR2인 경우)
WHERE emp_no = 12345

-- ✅ 좋은 예
WHERE emp_no = '12345'
```

> 이유: 데이터 타입이 다르면 오라클 내부적으로 컬럼을 형변환(`TO_NUMBER` 등)하므로 인덱스가 깨집니다.

### 부정형 조건(!=, NOT IN) 자제하기

인덱스는 "특정 값을 빨리 찾기" 위한 지도입니다. "~가 아닌 것"을 찾을 때는 인덱스가 효율적으로 작동하지 않으므로, 가급적 긍정형(`=`, `IN`)으로 표현하세요.

## 10. 조인과 집합 연산 최적화

### IN vs EXISTS 구별해서 쓰기

서브쿼리의 결과 집합이 크다면 EXISTS를 사용하는 것이 유리합니다. EXISTS는 조건에 맞는 **첫 번째 행을 찾는 즉시 검색을 멈추기** 때문입니다.

### UNION ALL을 우선 고려하기

UNION은 두 결과 집합을 합친 후 중복 제거를 위해 내부적으로 정렬(Sort) 작업을 거칩니다. **중복 가능성이 없거나 허용해도 된다면 정렬 과정이 없는 UNION ALL**을 쓰는 것이 훨씬 빠릅니다.

---

## 11. 데이터 조회 범위 최소화 (심화)

### ROWNUM 또는 페이징 처리 활용

화면에 10건만 보여주면 되는데 수만 건을 조회할 필요가 없습니다.
- 오라클 12c 이상: `FETCH FIRST n ROWS ONLY`
- 이전 버전: `WHERE ROWNUM <= n`

### EXISTS 활용

단순 데이터 존재 여부만 체크할 때는 IN보다 EXISTS가 서브쿼리 수행을 중간에 멈출 수 있어 유리할 때가 많습니다.

### COUNT(컬럼) 활용

특정 인덱스 컬럼만 카운트하면 `COUNT(*)` 대비 더 효율적일 수 있습니다.

---

## 12. 함수(Function) 사용 줄이기

### 사용자 정의 함수(UDF) 남용 금지

WHERE절이나 SELECT절에 개발자가 만든 PL/SQL 함수를 넣으면, **결과 행(Row)의 수만큼 함수가 매번 호출(Context Switching)** 되어 성능이 궤멸적으로 느려질 수 있습니다.

가급적 `DECODE`, `CASE`문이나 `JOIN`으로 해결하는 것이 좋습니다.
