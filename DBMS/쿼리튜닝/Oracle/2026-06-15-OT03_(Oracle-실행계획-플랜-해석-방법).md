## 1. 핵심 요약

오라클 실행계획 결과 표를 읽을 때 핵심은 **"안쪽(들여쓰기 깊은 곳)에서 바깥쪽으로, 위에서 아래로"** 실행 순서를 파악하고, **TABLE ACCESS FULL·Predicate Information·조인 방식** 세 가지를 집중 점검하는 것입니다.

---

## 2. 실행계획 기본 구조 예시

```
--------------------------------------------------------------------------------------
| Id  | Operation                    | Name   | Rows  | Bytes | Cost (%CPU)| Time     |
--------------------------------------------------------------------------------------
|   0 | SELECT STATEMENT             |        |     1 |    37 |            |          |
|   1 |  TABLE ACCESS BY INDEX ROWID | EMP    |     1 |    37 |            |          |
|* 2 |   INDEX UNIQUE SCAN          | PK_EMP |     1 |       |            |          |
--------------------------------------------------------------------------------------

Predicate Information (identified by operation id):
---------------------------------------------------
   2 - access("EMPNO"=7788)
```

---

## 3. 실행 계획 해석 순서 (가장 중요)

실행 계획은 위에서 아래로 그냥 읽으면 안 됩니다. 옵티마이저의 실행 순서는 **"안쪽에서 바깥쪽으로, 위에서 아래로"** 가 기본 원칙입니다. 즉, **들여쓰기(Indent)가 가장 깊게 들어간 번호가 1순위로 실행**됩니다.

위 예시로 순서를 찾아보면:

1. **Id 2 (INDEX UNIQUE SCAN)**: 가장 안쪽으로 들여쓰기가 되어 있으므로 **가장 먼저 실행**됩니다. PK_EMP 인덱스에서 조건에 맞는 Rowid(주소록)를 찾습니다.
2. **Id 1 (TABLE ACCESS BY INDEX ROWID)**: 인덱스에서 찾은 주소록을 가지고 실제 EMP 테이블의 데이터 블록에 접근합니다.
3. **Id 0 (SELECT STATEMENT)**: 최종 결과를 사용자에게 반환합니다.

> 💡 **쉽게 기억하는 팁**: 형제(들여쓰기 깊이가 같은) 간에는 **위에 있는 것**이 먼저 실행되고, 부모-자식(들여쓰기 깊이가 다른) 간에는 **자식(안쪽)** 이 먼저 실행됩니다.

## 4. 핵심 칼럼 항목별 의미

| 칼럼명 | 의미 | 튜닝 시 체크 포인트 |
|--------|------|---------------------|
| **Id** | 고유 단계 번호 | 앞에 `*` 표시가 붙어 있다면(예: `* 2`), 하단의 `Predicate Information`에서 구체적인 조건식을 확인해야 합니다. |
| **Operation** | 데이터를 가져오는 방식 | `FULL SCAN`인지 `INDEX SCAN`인지, 조인 방식(`HASH JOIN`, `NESTED LOOPS`)이 무엇인지 결정되는 가장 중요한 항목 |
| **Name** | 사용된 오브젝트 이름 | 어떤 테이블이나 어떤 인덱스가 사용되었는지 명시 |
| **Rows (Cardinality)** | 옵티마이저가 예상한 건수 | 이 단계에서 몇 건이 처리될지 예측한 수치. 실제 데이터 양과 터무니없이 다르면 **통계정보 갱신** 필요 |
| **Cost** | 쿼리 수행 비용 | 옵티마이저가 계산한 상대적인 '고생 수치'. 낮을수록 좋은 쿼리 |

## 5. 튜닝할 때 반드시 눈여겨봐야 할 3가지

### ① TABLE ACCESS FULL 확인

데이터가 수백만 건인데 이 메시지가 보인다면 **테이블 전체를 덤핑하며 읽고 있다**는 뜻입니다.
- 인덱스가 없거나
- 인덱스가 있어도 조건절 오류(예: 인덱스 컬럼 가공)로 인해 인덱스를 타지 못하는 상태일 확률이 높습니다.

### ② Predicate Information (조건절 정보)

실행 계획 맨 밑에 나오는 상세 설명입니다.

| 구분 | 의미 | 판단 |
|------|------|------|
| **Access** | 인덱스를 콕 집어서 효율적으로 타고 들어간 조건 | ✅ 아주 좋음 |
| **Filter** | 인덱스를 못 타거나, 데이터를 일단 다 가져온 다음 메모리에서 걸러낸 조건 | ⚠️ 비효율적일 수 있음 |

### ③ 조인 방식 (Join Method) 확인

두 개 이상의 테이블을 엮을 때 어떤 방식을 썼는지 확인합니다.

| 조인 방식 | 설명 | 유리한 상황 |
|-----------|------|------------|
| **NESTED LOOPS** | 한 건씩 순차적으로 조인 | 소량 데이터 처리, 인덱스가 잘 잡혀있을 때 |
| **HASH JOIN** | 양쪽 테이블을 메모리에 올려놓고 연결 | 대용량 데이터 처리 시 유리 |
