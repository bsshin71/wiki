## 1. 핵심 요약

오라클 19c SQL 튜닝은 **문제 SQL 식별 → 실행계획·통계 분석 → 원인 진단 → 개선 적용 → 검증** 의 순환 구조로 진행됩니다. 각 단계를 건너뛰면 엉뚱한 곳을 고치게 되므로 순서를 지키는 것이 중요합니다. 19c에서는 SQL Monitor, SQL Plan Baseline(SPM), SQL Tuning Advisor 등 강력한 자동화 도구를 활용할 수 있습니다.

## 3. 1단계 — 문제 SQL 식별

"무엇이 느린가"를 **추측 대신 객관적 도구**로 찾아야 합니다.

**AWR·ASH 리포트 활용:**
- `awrrpt.sql` / `ashrpt.sql` 로 인스턴스 전체에서 Elapsed Time, CPU Time, Buffer Gets, Disk Reads 상위 SQL을 식별

**실시간 직접 조회:**

```sql
SELECT sql_id, executions,
       elapsed_time/1e6                                          ela_sec,
       elapsed_time/DECODE(executions,0,1,executions)/1e6       ela_per_exec,
       buffer_gets, disk_reads, sql_text
FROM   v$sqlarea
ORDER  BY elapsed_time DESC
FETCH FIRST 20 ROWS ONLY;
```

> ⭐ **중요한 판단 기준**: 단순 총 수행시간이 아니라 **실행당 비용(per execution)** 과 **실행 횟수** 를 함께 봐야 합니다.
> - 한 번에 느린 SQL인지
> - 가볍지만 너무 자주 호출되는 SQL인지
> 에 따라 처방이 완전히 달라집니다.

## 4. 2단계 — 실행계획 분석

대상 SQL이 정해지면 옵티마이저가 실제로 어떤 계획을 선택했는지 확인합니다.

> ⚠️ `EXPLAIN PLAN`은 "예상 계획"일 뿐, 실제 실행과 다를 수 있습니다. **실제 실행된 계획과 실측치**를 봐야 합니다.

### 실측 통계를 포함한 Real 실행계획 확인

```sql
-- 실제 실행된 커서의 계획 + 실측 통계 확인
SELECT * FROM TABLE(
  DBMS_XPLAN.DISPLAY_CURSOR('&sql_id', NULL,
    'ALLSTATS LAST +PEEKED_BINDS +OUTLINE')
);
```

`ALLSTATS LAST` 옵션을 쓰려면 해당 세션에서 `STATISTICS_LEVEL = ALL`로 실행하거나 SQL에 `/*+ GATHER_PLAN_STATISTICS */` 힌트를 넣어 실행한 뒤 조회합니다. 이렇게 하면 **E-Rows(예상)와 A-Rows(실제)** 를 나란히 볼 수 있는데, **이 둘의 괴리가 튜닝의 출발점**입니다.

### 계획 점검 핵심 포인트

| 점검 항목 | 의심 기준 |
|-----------|-----------|
| 카디널리티 오추정 | E-Rows와 A-Rows 10배 이상 차이 |
| 불필요한 Full Table Scan | 대용량 테이블에 TABLE ACCESS FULL |
| 부적절한 조인 방식 | 소량 조인에 Hash Join, 대량 조인에 Nested Loop |
| 잘못된 조인 순서 | 드라이빙 테이블 건수가 너무 많음 |
| 인덱스 미사용 | 인덱스가 있는데도 INDEX SCAN 미발생 |

### 19c SQL Monitor (가장 강력)

5초 이상 걸리거나 병렬 수행된 SQL은 자동으로 모니터링되며, 각 실행 단계별로 시간이 어디서 소모되는지 시각적으로 보여줍니다.

```sql
SELECT DBMS_SQLTUNE.REPORT_SQL_MONITOR(
         sql_id => '&sql_id', type => 'TEXT') FROM dual;
```

## 5. 3단계 — 통계정보 점검

잘못된 실행계획의 **가장 흔한 원인**은 부정확하거나 오래된 통계입니다.

### 통계 상태 확인

```sql
SELECT table_name, num_rows, last_analyzed, stale_stats
FROM   user_tab_statistics
WHERE  table_name = '&table_name';
```

`STALE_STATS = YES`이거나 `LAST_ANALYZED`가 오래됐다면 통계를 재수집합니다.

### 통계 재수집

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'SCOTT',
    tabname          => 'EMP',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE   -- 인덱스 통계도 함께 수집
  );
END;
/
```

> **주의사항**:
> - 데이터가 균등하지 않은 컬럼(특정 값에 쏠림이 있는 컬럼)에는 **히스토그램**이 필요합니다
> - 반대로 바인드 변수와 함께 쓰는 컬럼의 히스토그램은 오히려 **계획 불안정**을 유발할 수 있으니 상황을 봐서 판단합니다

## 6. 4단계 — 원인 진단

2~3단계에서 모은 정보로 진짜 원인을 찾습니다. 자주 나오는 패턴:

| 패턴 | 원인 | 진단법 |
|------|------|--------|
| **카디널리티 오추정** | 통계 문제, 함수 씌워진 조건, 복합 컬럼 간 상관관계 미반영 | E-Rows vs A-Rows 비교 |
| **인덱스 미사용** | 컬럼에 함수·연산 적용, 데이터 타입 암묵적 변환, 인덱스 선두 컬럼 조건 부재 | Predicate Information 확인 |
| **부적절한 조인 방식** | 선행 단계의 카디널리티 오추정에서 비롯됨 | 조인 순서·방식 확인 |

## 7. 5단계 — 개선 적용

원인에 맞는 처방을 적용합니다. **근본 원인(통계·SQL 구조)부터 손대고, 힌트나 SQL 고정은 그다음입니다.**

### SQL 자체 수정 (최선)

- 인덱스를 탈 수 있도록 조건절 수정 (함수 제거, 타입 일치)
- 불필요한 서브쿼리나 중복 조인 제거
- 필요한 컬럼만 조회
- 조건+정렬을 모두 커버하는 복합 인덱스 추가

### SQL 수정 불가 시 — 계획 외부 고정 방법

**SQL Plan Baseline(SQL Plan Management, SPM)** — 가장 권장하는 방식. 검증된 계획만 사용하도록 강제해 계획 변경에 따른 성능 저하를 막습니다.

```sql
-- 커서 캐시에서 baseline으로 계획 고정
DECLARE
  l_plans PLS_INTEGER;
BEGIN
  l_plans := DBMS_SPM.LOAD_PLANS_FROM_CURSOR_CACHE(sql_id => '&sql_id');
END;
/
```

| 방법 | 설명 | 권장도 |
|------|------|--------|
| **SQL Plan Baseline (SPM)** | 검증된 계획만 사용하도록 강제 | ⭐⭐⭐ 가장 권장 |
| **SQL Profile** | 통계 보정 없이 옵티마이저 추정을 교정 | ⭐⭐ |
| **SQL Patch** | 특정 힌트만 주입 | ⭐⭐ |
| **힌트 직접 삽입** | 즉효성 있지만 운영 부담 큼 | ⭐ 신중히 사용 |

## 8. 6단계 — 자동화 도구 활용 (19c)

직접 분석이 어려울 때 19c의 어드바이저를 활용합니다.

### SQL Tuning Advisor

대상 SQL을 분석해 통계 보강, SQL Profile, 인덱스, SQL 재작성 등을 권고합니다.

```sql
DECLARE
  l_task VARCHAR2(64);
BEGIN
  l_task := DBMS_SQLTUNE.CREATE_TUNING_TASK(sql_id => '&sql_id');
  DBMS_SQLTUNE.EXECUTE_TUNING_TASK(l_task);
  DBMS_OUTPUT.PUT_LINE(l_task);
END;
/

-- 결과 조회
SELECT DBMS_SQLTUNE.REPORT_TUNING_TASK('&task_name') FROM dual;
```

| 어드바이저 | 담당 영역 |
|-----------|-----------|
| **SQL Tuning Advisor** | 개별 SQL 분석·개선 권고 |
| **SQL Access Advisor** | 인덱스·MV 등 접근 구조 전반 |
| **SQL Performance Analyzer (SPA)** | 변경 전후 성능 영향 비교 |

> ⚠️ 어드바이저의 권고는 맹신하지 말고 **직접 검증**하는 것이 좋습니다.

---

## 9. 7단계 — 검증

개선 후 반드시 효과를 측정합니다.

**검증 항목:**
- 동일 조건에서 실행계획이 의도대로 바뀌었는지
- A-Rows와 E-Rows 괴리가 줄었는지
- Buffer Gets·Elapsed Time이 실제로 감소했는지 (튜닝 전과 비교)
- **부작용 점검**: 한 SQL을 고쳤더니 다른 SQL이 느려지는 경우(인덱스 추가로 인한 DML 부하 증가 등)도 함께 점검

---

## 10. 실무 팁 (19c 특화)

19c에서는 **Adaptive Plans, Automatic SQL Plan Management, Real-Time Statistics** 같은 자동화 기능이 강화됐습니다.

- 도움이 될 때가 많지만 **예측 불가능한 계획 변동**을 일으키기도 합니다
- 운영 안정성이 중요한 시스템에서는 **SQL Plan Baseline으로 계획을 고정**해 변동을 통제하는 전략이 자주 쓰입니다
